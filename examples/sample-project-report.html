项目推进 / 合伙协作 / MVP 收口

合伙创业项目：为什么越开会越推不动

目标是 6 周内上线 MVP，但产品、技术和创始人在需求会上连续拉扯两周，大家都很努力，事情却始终没有收口。这份报告要回答的不是“谁对谁错”，而是现在到底卡在哪、先砍哪、谁拍板、怎么动。

当前别再一边赶时间一边舍不得砍需求了，先收口 MVP，再定唯一拍板人，再跑一轮真验证。

I / Fixed Core

核心判断区

先把目标、阶段、主问题和当前第一要务压稳，后面的所有模块都围绕这一层展开。

固定骨架

用户目标

6 周内上线 MVP，拿到第一批付费用户，而不是继续在内部打转。

当前阶段

项目已经没有条件继续“边讨论边发散”了，旧办法失灵是因为团队还在用探索期的心态处理收口期的问题。

核心问题

不是大家不努力，而是主路线没收口、主责没归位、创始人又总想把两边都保住。

当前第一要务

先把 MVP 定义砍清楚，并明确一个对上线标准负责的人，不再让会议继续无限翻案。

II / Relation Graph

关系图模块

当局面真正难在“谁影响谁、谁卡谁、谁依赖谁”时，先把结构图画清，再进入路线判断。

复杂关系优先

人物与事项关联图

这个局的难点不是谁更有道理，而是创始人、产品、技术、上线动作和现金流彼此怎么卡住了彼此。节点、关系线都可以点击。

节点可点 关系线可点 先看中心点

使用方式

  • 先看中心节点，再看四条关键关系线。
  • 优先保留 5 到 9 个关键节点，不要把噪音也塞进图里。
  • 点击节点或关系线，可以展开更细的结构信息。

关系索引

  • 创始人 -> MVP 收口失败 既是推动力，也是“什么都想保”的放大器。
  • 产品 / 技术双线 -> MVP 收口失败 两条合理诉求在没有统一标准时，持续互相抵消。
  • MVP 收口失败 -> 上线动作 只要范围不收口，执行就会一直漂。
  • 3 个月现金流 -> MVP 收口失败 每一次返工，都在直接消耗项目剩余寿命。

III / Main Route

建议路线区

先定路线，再写前提，最后压成关键动作。这里负责“当前主路线”，不是路线大全。

固定骨架

先砍范围，再定拍板人，再跑一轮真验证

当前最优路线不是继续把需求会开得更长，而是先把 MVP 压缩成一条最小闭环，再用真实反馈替代内部空转。

成立前提

创始人接受先保一个目标；产品和技术都接受统一上线标准；团队同意本周只保留一次拍板会。

路线说明

先把“要什么”收口，再把“谁拍板”定死，最后用真实用户反馈检验路线，而不是继续在内部靠想象拉扯。

1

48 小时内冻结范围

把需求分成“必须有 / 可以后补 / 暂不做”三层，当场定掉，不留会后再翻案。

2

指定唯一拍板人

明确谁对上线标准负责，其他人可以提意见，但不能反复重新开题。

3

一周内跑真验证

拿一个最小可用版本给真实用户试，用反馈判路线，不再靠内部脑补用户。

IV / Immediate Actions

下一步行动区

把近期动作、观察点和风险提醒压成闭环，不在这里展开长期执行计划。

固定骨架

近期动作

今天就约一场只做收口、不做脑暴的 45 分钟会，并在会后把冻结范围发全员确认。

观察点

如果争论明显减少、排期不再来回改、技术任务开始连续推进，说明主路线开始收拢；反之就要回头查拍板权是否还在漂。

风险提醒

最危险的不是冲突，而是继续用“先都做”拖延真正的取舍，把团队拖进无限返工。

VI / Timeline

时间线模块

阶段变化明显、事件顺序会影响判断时保留。重点写“它怎样改变局面”，不要只是流水账。

按需保留

第 1 周 / 目标刚定

团队把目标定成“尽快上线 MVP”

当时大家都认同要快，但没有把“什么算 MVP”同步定义清楚，后面的争论种子从这里就埋下了。

第 2-3 周 / 需求不断加项

产品想把体验讲圆，技术开始担心返工

两边都不是没道理，但因为没有统一上线标准，讨论逐渐从“怎么做”变成“谁来定义什么叫够”。

第 4 周 / 冲突升级

创始人每次都试图“两边都保”

结果不是冲突被解决，而是冲突被延后。最后人人看起来都在忙，实际没有一条主路线真正收口。

VII / Structure

结构拆解模块

表层问题和深层问题不同，或需要区分主次问题时保留。它负责补层次，不负责重复核心判断。

按需保留

核心问题

不是谁态度不好，而是 MVP 定义和拍板机制都没有收口。

次级问题

会议过长、表达方式冲、任务拆得不够细，这些都会增加成本，但它们不是主入口。

暂不优先处理

品牌打磨、非核心功能、上线后才能验证的高级体验项，本轮都不该抢入口。

VIII / Route Comparison

路线比较模块

存在两条及以上可行路线时保留。这里负责比较备选，而不是把主路线再重写一遍。

按需保留

继续双线并行

适用前提

现金流充裕，且团队能承受继续探索 1 到 2 个月。

关键动作

同时保留体验完整性和技术稳态目标，边做边调。

主要风险

范围继续膨胀，返工越来越多，最后谁都累但还是不上线。

预期结果

表面看起来很积极，实际上大概率继续原地打转。

先收口 MVP 再上线

适用前提

团队接受先保主闭环，不追求一步到位。

关键动作

冻结范围、定唯一拍板人、做一轮真实用户验证。

主要风险

短期会有人觉得“做得太少”，心理上不舒服。

预期结果

最快把局面从空转拉回推进，是当前最稳的一条路。

暂停两周先补底层

适用前提

现有代码或流程已经脆到根本无法承接上线动作。

关键动作

先补关键基础设施，再恢复交付。

主要风险

容易变成新的拖延借口，错过市场窗口。

预期结果

只有在底层真的撑不住时才成立，不是默认选项。

IX / Resources

资源与约束模块

资源配置、承受能力、约束条件会改变路线时保留。

按需保留

关键资源

创始人的拍板权、技术负责人的交付能力、第一批种子用户窗口。

当前约束

现金流只够 3 个月，团队没有继续无限返工的预算。

主要风险

每多一轮返工，试错窗口和团队士气都会一起被吃掉。

X / Communication

沟通与协商模块

关键突破点在表达对象、说法设计和时机安排时保留。

按需保留

沟通对象与目标

对创始人：只保一个目标；对产品和技术：统一上线标准；对全员：本周不再反复开题。

核心信息与时机

会前发需求三分层简表；会上只做取舍，不做脑暴；会后当场定稿，不给“回去再想想”留口子。

XI / Execution

执行计划模块

重点在推进落地、安排节奏和反馈调整时保留。它负责拉长节奏，不重复“下一步行动区”。

按需保留

24 小时

完成范围三分层、唯一拍板人确认，并把上线标准写成一页纸。

7 天

做出最小闭环版本，锁住需求变更，跑一轮真实用户试用，看有没有人愿意继续用或付费。

30 天

根据用户反馈和付费信号，决定是继续扩功能，还是继续压缩路线，不再靠内部好恶拍脑袋。
