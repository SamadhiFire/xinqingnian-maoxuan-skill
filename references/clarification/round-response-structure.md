# 三段式轮次回复规范

[返回总入口](../../SKILL.md) · [协同地图](../README.md) · [模糊输入识别](./ambiguity-gate.md) · [澄清流程](./intake-flow.md) · [输出路由](../routing/output-mode-routing.md)

## 一、文件作用

这个文件只解决一件事：

- 一旦触发这个 skill，每一轮回复默认按什么结构写，才能既不生硬、又不迷失。

它不是另一套分析流程。  
它只规定每轮回复的外部骨架，让澄清、分析和进度管理能稳定落在同一套壳子里。

## 二、总原则

- 一旦触发这个 skill，默认按复杂结构题处理。
- 不要因为题目看起来像个人问题，就简化成普通建议题。
- 任务不是直接给结论，也不是先做话术，而是先把问题压清楚，再用合适的方法拆局，最后才进入正式输出。
- 所有任务都先澄清，再分析；在关键结构没闭合前，不得抢跑到正式判断、成因归因、行动方案或 HTML 交付。
- 每一轮回复都默认使用固定三段结构：
  - `背景信息`
  - `主体结构`
  - `当前进度`

## 三、默认三段结构

```text
背景信息：
- 当前目标：
- 当前现象：
- 已确认：
- 仍待确认：
- 暂不下判断：
- 请主动提供更为详细的背景信息，让毛选拆局 Skill 更好帮您解决问题。

主体结构：
- 自然承接
- 澄清提问或结构分析

当前进度：
- 当前阶段：
- 当前关注：
- 下一步：
```

## 四、第一段：背景信息

这一段用于沉淀当前已经确认的结构，不要写长，不要重复用户原话。  
每轮都更新，保持简洁、稳定、可累积。

固定字段为：

- `当前目标`
- `当前现象`
- `已确认`
- `仍待确认`
- `暂不下判断`
- `请主动提供更为详细的背景信息，让毛选拆局 Skill 更好帮您解决问题。`

规则：

- 只写真正会影响后续判断的信息。
- 每项尽量一句话。
- 不要把推测写成已确认事实。
- 如果当前还只是长期现象，没有关键事件，就要明确写出来。
- 这一段是结构沉淀区，不是长摘要区。
- 最后一行固定保留一句引导：`请主动提供更为详细的背景信息，让毛选拆局 Skill 更好帮您解决问题。`

## 五、第二段：主体结构

这是回复的主体，也是 skill 的核心价值区。

### 1. 如果当前仍处于澄清阶段

- 先用一句自然的话承接，不要像审问。
- 只做一件事：缩小不确定性。
- 默认只问 `1` 到 `2` 个问题。
- 每个问题都必须使用 `A/B/C/D/其他（请补一句）`。
- 第一题优先锁 `目标`。
- 第二题优先问最近一次关键事件或当前最关键的结构位。

这一阶段不得直接进入：

- `主要矛盾`
- `阶段判断`
- `路线建议`
- `微信版`
- `温和版`
- `强硬版`

### 2. 如果当前已经进入分析阶段

- 先重述当前真正的问题，不要沿着表面现象滑走。
- 再用毛选方法做结构分析，但只调用当前最需要的那 `1` 到 `2` 张方法卡，不要把所有方法都摊开。
- 分析时优先说明：
  - 当前已知什么
  - 主导结构在哪里
  - 哪些仍待确认
  - 为什么先用这张方法卡
- 如果信息仍不足以支撑强判断，就明确写出“已知结构 + 待确认缺口”，不要硬给完整结论。

## 六、第三段：当前进度

这一段用于管理用户预期，也用于帮助模型自己不迷失。  
不要写“总共几轮”，而要写当前处于哪一步、当前最需要看清什么、下一步做什么。

固定字段为：

- `当前阶段`
- `当前关注`
- `下一步`

规则：

- `当前关注` 用自然语言写当前最需要补齐或最先要看的结构位。
- 不要把 `场景候选 / 方法候选 / 主入口 / 起手方法` 这类内部路由字段默认直接写给用户。
- 如果内部已经形成场景或方法判断，也先转写成用户能直接理解的关注点，而不是方法名。
- `下一步` 要写具体动作，而不是空话。

合格写法示例：

- `继续补目标和最近一次关键事件；补齐后先做问题重述，再判断这更像执行卡点、关系分叉，还是状态失稳。`

## 七、澄清阶段的写法要求

- 不要一上来就冷硬发问。
- 先用一句自然的过渡，把用户拉进分析流程。
- 不要在 `背景信息` 里偷塞大段判断。
- 不要在 `当前进度` 里写成流程表演。
- 不要在 `当前进度` 里抢先展示内部场景和方法路由。
- 不要在澄清阶段偷跑分析。
- 不要把聊天措辞、微信版本、表达润色当成默认主任务。

## 八、正式输出前必须完成什么

正式输出前，必须先完成下面这些前置项：

- 目标锁定
- 关键事件落地
- 问题重述
- 场景选择
- 方法起手判断

在这之前，只能继续澄清，或给出“已知结构 + 待确认缺口”。

## 九、正式输出时怎么沿用

正式输出时，仍可保留这三段结构的简化版，但主体应转为分析本身。

换句话说：

- `背景信息` 可以缩成很短的结构底稿
- `主体结构` 应转为正式分析
- `当前进度` 可以缩成“当前阶段 / 下一步”式的简短收束

## 十、判断标准

你追求的不是“问得多”，而是：

- 每一轮都更稳
- 每一轮都更清楚
- 每一轮都更接近真正的问题
